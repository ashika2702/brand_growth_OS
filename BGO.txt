Section 1 — What Is BGO

The One Paragraph Explanation
BGO is a living autonomous intelligence system that manages a business's entire marketing operation. It watches, thinks, creates content, manages leads, tracks SEO, monitors competitors, manages reviews, runs campaigns, and learns — 24 hours a day, 7 days a week.
There are two modes: Manual Mode (the Account Manager uses the platform like a cockpit — clicks, configures, approves) and Autopilot Mode (BGO Intelligence operates the system autonomously — navigating screens, creating content, making decisions — and asks the human only when it needs judgment or approval).
The human is always the governor. Nothing that touches the outside world — publishing, sending, posting, spending money — happens without a human gate. BGO prepares everything. The human releases it.

The Real Business Problem BGO Solves
A marketing agency managing 8 clients today uses: a spreadsheet for leads, Google Docs for briefs, a separate SEO tool, a separate ads tool, a separate scheduling tool, a separate reporting tool. None of them talk to each other. An insight from the ads tool never reaches the content brief. A competitor move spotted on Monday is forgotten by Friday. The account manager spends 70% of their time on operational tasks that add no strategic value.
BGO replaces all of that. One platform. Everything connected. AI agents handle the monitoring, drafting, scheduling, scoring, and reporting. The account manager focuses on judgment and client relationships.

The Three Users
User	Who They Are	What They Do
Account Manager	Marketing team member — manages clients day to day	Uses Command Centre every morning, manages CRM, approves content, runs Content Tap, uses AI Assist panel, sets autopilot per client
Client	The business owner — plumber, café, kitchen company	Logs into Client Portal on phone. Approves content, reads reports, sees lead count and keyword rankings. Simple view. No jargon.
Founder / Admin	Thiru — sees everything	Oracle view (all client margins), all agent logs, all AI costs, platform settings, white-label config, override history

The Two Operating Modes
MODE 1 — MANUAL (AM drives)
The AM uses the platform directly. Clicks to generate a brief. Drags leads through the pipeline. Approves content. Configures campaigns. The AI assists and suggests — but the AM initiates every action.
The Guardian Agent watches every action the AM takes and surfaces smart suggestions in real time. It never blocks. It always informs. Think of it as the most experienced colleague sitting next to you.

MODE 2 — AUTOPILOT (BGO Intelligence drives)
BGO Intelligence takes the controls. It navigates the platform autonomously — opening modules, reading data, creating content, editing video, managing campaigns, drafting responses. The AM watches it work on Mission Control.
Every action streams to the Live Action Feed. Every decision is logged with reasoning. Nothing that touches the outside world executes without a Human Gate — the AM approves, provides information, or overrides.
Autopilot is per-client and per-task. One client on full autopilot, another on manual, another on hybrid. The AM sets it.


Section 2 — The Two Laws

THE TWO LAWS — Read this before writing any AI feature. This is the most important rule in the codebase.
LAW 1: Every single Claude or Qwen call must include getBrainContext(clientId) in the system prompt.
LAW 2: Every single Claude or Qwen call must include getRoleContext(userId, moduleName) in the system prompt.
These two functions together form the system prompt for every AI call. If either is missing, the AI output will be generic, wrong, and off-brand. This is the most common bug. Check both on every AI feature you build.
The pattern is always: system = getRoleContext(userId, module) + newline + newline + getBrainContext(clientId)

getBrainContext() — What It Is and What It Returns
The Business Brain is a detailed intelligence file about each client. getBrainContext() reads it from the database and returns it as a string that Claude reads before doing anything for that client. With a rich Brain, Claude knows exactly who the client is, what they sell, how they speak, what their customers care about, and what has worked in the past. Without it, Claude is guessing.

// lib/brain.ts
export async function getBrainContext(clientId: string): Promise<string> {
  const { data: brain } = await supabase
    .from('business_brain')
    .select('*')
    .eq('client_id', clientId)
    .single();

  return `
  ## BUSINESS: ${brain.business_name}
  Industry: ${brain.industry} | Location: ${brain.service_areas}
  Unique value: ${brain.unique_value_proposition}

  ## PERSONAS
  ${brain.personas.map(p => `
    ${p.name} (${p.age}): Pain = ${p.pain_points}. Proof type = ${p.proof_type}.`,
  ).join('')}

  ## OFFERS
  ${brain.offers.map(o => `${o.name}: ${o.price_range} — ${o.margin_tier} margin`).join('\n')}

  ## VOICE GUIDE
  Tone: ${brain.voice_guide.tone_adjectives.join(', ')}
  Never say: ${brain.voice_guide.words_to_avoid.join(', ')}
  Sample approved content: '${brain.voice_guide.sample_approved[0]}'

  ## WHAT WORKS FOR THIS AUDIENCE (learned from performance data)
  ${brain.content_patterns?.works ?? 'Not yet collected — grows after 30 days'}

  ## WHAT DOES NOT WORK
  ${brain.content_patterns?.avoid ?? 'Not yet collected'}

  ## COMPETITIVE LANDSCAPE
  ${brain.competitor_intel?.summary ?? 'Not yet scanned — A2 runs Sunday 11pm'}
  `;
}

// BUSINESS: Sydney Kitchen Co | Industry: Kitchen Renovation | Location: Western Sydney

// VOICE: warm, direct, no jargon. Never say: 'leverage', 'seamless', 'cutting-edge'

getRoleContext() — What It Is and What It Returns
This function returns the AI persona and instructions based on who is asking and which module they are in. Alex is the Account Manager persona. Max is the Content Strategist. Quinn is the client-facing assistant. Nova is the SEO and data analyst. Oracle is the founder-level strategist. The AI sounds different in each module because it is playing a different role.

// lib/roles.ts
export async function getRoleContext(userId: string, module: string): Promise<string> {
  const user = await getUser(userId);
  const persona = PERSONA_MAP[module] ?? PERSONA_MAP['default'];
  return `You are ${persona.name}, ${persona.role}.
  You are helping ${user.name} (${user.role}) with ${module}.
  ${persona.instructions}
  Always be direct, specific, and actionable. Never generic.`;
}

const PERSONA_MAP = {

// lib/llm.ts — The unified router. Copy this exactly.
import LiteLLM from 'litellm';

export type TaskType = 'classify'|'extract'|'score'|'summarise_bulk'  // → Qwen
                     | 'analyse'|'synthesise'|'brief'|'recommend'|'create'  // → Claude
                     | 'deep_reason'|'simulate'|'evaluate'|'plan';  // → NemoClaw

const MODEL_MAP: Record<TaskType, string> = {
  classify:       'ollama/qwen2.5:14b',
  extract:        'ollama/qwen2.5:14b',
  score:          'ollama/qwen2.5:14b',
  summarise_bulk: 'ollama/qwen2.5:14b',
  analyse:        'claude-sonnet-4-20250514',
  synthesise:     'claude-sonnet-4-20250514',
  brief:          'claude-sonnet-4-20250514',
  recommend:      'claude-sonnet-4-20250514',
  create:         'claude-sonnet-4-20250514',
  deep_reason:    'ollama/nemoclaw:latest',
  simulate:       'ollama/nemoclaw:latest',
  evaluate:       'ollama/nemoclaw:latest',
  plan:           'ollama/nemoclaw:latest',
};

export async function llm(task: TaskType, prompt: string, system?: string) {
  const model = MODEL_MAP[task];
  const isLocal = model.startsWith('ollama');
  const response = await LiteLLM.completion({
    model,
    messages: [
      ...(system ? [{ role:'system', content:system }] : []),
      { role:'user', content:prompt }
    ],
    api_base: isLocal ? process.env.OLLAMA_BASE_URL : undefined,
  });
  return response.choices[0].message.content;
}

3.3 — The Universal Interaction Pattern
Every user action in BGO follows this exact pattern. Learn this once — you can build any feature.

USER CLICKS / SUBMITS SOMETHING IN THE UI
  │
  ▼
UI: show loading state (spinner on button, skeleton on cards)
  │
  ▼
fetch() → POST or GET /api/[module]/[action]
  │
  ▼
API ROUTE:
  1. Parse body (Zod validation)
  2. Check auth (supabase.auth.getSession())
  3. If data only: query Supabase → return JSON
  4. If AI needed:
       roleCtx  = await getRoleContext(userId, module)   ← LAW 1
       brainCtx = await getBrainContext(clientId)         ← LAW 2
       system   = roleCtx + '\n\n' + brainCtx
       result   = await llm(taskType, prompt, system)     ← LiteLLM routes
       save result to Supabase
       return result
  │
  ▼
UI: receives response → update state → re-render
  ├── Success: show result / toast('Success') / navigate
  └── Error:   toast('Error message') / show retry button

// BACKGROUND PATTERN (agents — no user trigger):

3.4 — Supabase Realtime — How Live Updates Work
// In any React component that needs live data:
useEffect(() => {
  const channel = supabase
    .channel('my-channel-' + clientId)
    .on('postgres_changes', {
      event:  '*',          // INSERT | UPDATE | DELETE | * (all)
      schema: 'public',
      table:  'notifications',
      filter: `client_id=eq.${clientId}`
    }, (payload) => {
      // payload.new = the new row, payload.old = previous row
      setNotifications(prev => [payload.new, ...prev]);
      toast.success(payload.new.title);
    })
    .subscribe();

  return () => supabase.removeChannel(channel);  // cleanup
}, [clientId]);

// This is how: agent runs at 6am → writes notification → AM's screen
// updates automatically without refresh. Real-time. Live.

// Lead Score Formula — 5 factors, configurable weights
function calculateLeadScore(lead, brain) {
  const personaMatch    = matchPersona(lead, brain.personas);     // 0-30 pts
  const sourceQuality   = SOURCE_SCORES[lead.source];             // 0-25 pts
  const behaviourScore  = lead.pages_visited * 3;                 // 0-20 pts
  const stageVelocity   = getStageVelocity(lead.stage_history);  // 0-15 pts
  const offerFit        = matchOffer(lead, brain.offers);         // 0-10 pts
  return personaMatch + sourceQuality + behaviourScore + stageVelocity + offerFit;
}

// SOURCE_SCORES:
// google_ads=22, referral=25, organic=18, meta_ads=20, qr_code=15, manual=10

// 80-100 = HOT (red card border, immediate notification to AM)

// UI: Text input field. Submit → agent continues.

// GATE TYPE 2: APPROVAL GATE
// When: Action needs human sign-off
// Example: 'Approve publishing this Instagram Reel at 6pm?'
// UI: Preview of action + [✓ Approve] [✎ Modify] [✕ Decline]

// GATE TYPE 3: THRESHOLD GATE
// When: Financial or reputational threshold exceeded
// Default thresholds: $200 spend change, 20% budget shift
// UI: Shows predicted impact + risk + options

// GATE TYPE 4: CONFIDENCE GATE
// When: Agent confidence < 60%
// Example: 'I am 48% confident. Want to review before I proceed?'
// UI: Shows what it knows, what it doesn't. AM can override (logged).

// DB TABLE: human_gates
// { id, agent_id, client_id, gate_type, question, context_json,
//   options_json, status:'pending'|'approved'|'declined'|'overridden',
//   resolved_by, resolved_at, resolution_note }

// OVERRIDE SYSTEM:
// AM can always override any gate. One click.
// Override logged: who, when, what was overridden, optional reason.
// Repeated overrides on same gate type: agent learns → adjusts threshold.

export async function POST(request: Request) {
  const { clientId, userId, userInput } = await request.json();

  // THE TWO LAWS — always both, always first:
  const roleCtx  = await getRoleContext(userId, 'content');  // ← change module

  const result = await llm('brief', userInput, system);       // → Claude

  await supabase.from('your_table').insert({ client_id:clientId, result });

}

useEffect(() => {
  const channel = supabase
    .channel('leads-' + clientId)
    .on('postgres_changes', {
      event:  'INSERT',
      schema: 'public',
      table:  'leads',
      filter: `client_id=eq.${clientId}`
    }, (payload) => {
      setLeads(prev => [payload.new, ...prev]);

    })
    .subscribe();
  return () => supabase.removeChannel(channel);
}, [clientId]);

Pattern 5 — Optimistic Kanban Drag
const handleDragEnd = async (event: DragEndEvent) => {
  const { active, over } = event;
  if (!over || active.id === over.id) return;

  const newStage = over.id as string;
  const leadId   = active.id as string;
  const prevStage = active.data.current.stage;  // save for rollback

  // 1. Update UI immediately (optimistic)
  setLeads(prev => prev.map(l => l.id===leadId ? {...l, stage:newStage} : l));

  // 2. Save to DB
  const { error } = await supabase.from('leads')
    .update({ stage:newStage, stage_updated_at:new Date() })
    .eq('id', leadId);

  // 3. If error: revert
  if (error) {
    setLeads(prev => prev.map(l => l.id===leadId ? {...l, stage:prevStage} : l));
    toast.error('Could not update. Please try again.');
  }

  // 4. Fire workflow event
  if (!error) await automation.emit(`lead.stage=${newStage}`, { leadId, clientId });
};

export async function POST(request: Request) {

}

The Rule Before Writing Any Feature
1. Does it read from or write to Supabase? → Check RLS is set. Users only see their own data.
2. Does it call AI? → Both getRoleContext() AND getBrainContext() must be in the system prompt. No exceptions.
3. Does it create an important event? → Call createNotification().
4. Does it change data that could update in the background? → Add Supabase Realtime subscription.
5. Does it trigger automation? → Call automation.emit('event.name', payload).
6. Is it an agent autonomous action? → Log to action_log. If it touches the outside world, create a Human Gate first.
